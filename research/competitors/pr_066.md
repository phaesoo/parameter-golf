## Key Code Changes

### Sliding Window Eval (biggest single lever)
- `EVAL_STRIDE=64` with `TRAIN_SEQ_LEN=4096` window
- Each token gets up to 4032 tokens of context vs baseline's ~512 average
- Windows advance by 64 tokens; only rightmost 64 are scored per window
- Uses `torch.compile(model.forward_logits)` for speed
- Eval takes ~170s (within 10-min eval budget)
- Purely eval-time trick — does not change trained model

### int6 Quantization
- Mixed precision: int6 [-32, 31] on 2D weight matrices
- fp16 passthrough on tied embedding + control tensors
- "Lazy packing": store 6-bit values in int8 containers, let zstd-22 compress unused bits
- Avoids bit-packing complexity, similar compression ratio

### Wider MLP
- MLP_MULT=3.0, hidden=1536 (vs baseline 2x/1024)
- 1536 is 64-aligned for H100 matmul tile utilization
- Made possible by int6 saving ~4MB artifact space

### Optimizer Changes
- MUON_MOMENTUM=0.99 (up from 0.95)
- MATRIX_LR=0.02, SCALAR_LR=0.02 (halved) — explicitly to reduce quantization gap
- TRAIN_BATCH_TOKENS=393216 (3/4 of default) — more update steps per wallclock
- WARMDOWN_ITERS=3000, MUON_MOMENTUM_WARMUP_STEPS=1500

## Novel Ideas

1. **Sliding window eval is the biggest single lever (+0.032).** Post-quant sliding BPB (1.1661) is BETTER than pre-quant fixed-chunk BPB (1.1715). The eval trick alone is worth more than all architecture changes combined.
2. **int6 as size budget enabler, not quality technique.** The point of int6 isn't preserving quality — it's freeing bytes for a wider MLP.
3. **Lower LR reduces quantization gap.** Halving LR makes weights more quantization-friendly. QAT was tried and found redundant once quant gap was already near zero.
4. **H100 tile alignment matters.** MLP_HIDDEN=992 caused 8ms/step overhead vs 1536 (64-aligned).
5. **Built using an "AutoResearch agent"** that scraped all open PRs via `gh pr diff` and composed best techniques.

## Takeaways

- Sliding window eval is almost mandatory for competitive scores
- int6 + zstd is strictly better than int8 + zlib for this challenge
- The model architecture itself is barely changed from baseline (just wider MLP)
- Most improvement comes from eval tricks + quantization scheme, not architecture innovation
